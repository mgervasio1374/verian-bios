### 1. FULL SYSTEM ARCHITECTURE

#### Application Boundary

Verian BIOS is a **modular monolith** hosted as a single Next.js application. It is NOT split into microservices in Phase 1ŌĆō2. Service boundaries are enforced by code organization (modules, services, repositories), not network topology. This allows fast iteration now, with extraction later if needed.

#### Service Boundaries (Logical)

| Module | Owns | Calls |
|--------|------|-------|
| `platform` | tenants, workspaces, memberships, roles, entitlements | auth |
| `crm` | companies, contacts, leads, accounts, opportunities, activities, notes, tasks | platform |
| `workflow` | system_events, workflow_runs, job_executions, approval_requests | platform, crm |
| `intelligence` | scores, recommendations, prompt_configs, scoring_configs | platform, crm |
| `artifacts` | artifacts, versions, links, evidence_records | platform, storage |
| `messaging` | sender_identities, templates, drafts, sends, events | platform, resend |
| `config` | industry_profiles, pipeline_stages, dashboard_configs, policy_rules | platform |

#### Data Ownership
- **Postgres (Supabase)** = source of truth for all records
- **Inngest** = source of truth for workflow/job execution state
- **Supabase Storage** = source of truth for file blobs
- **Resend** = source of truth for delivery events (mirrored into `email_events`)

#### Runtime Flow
1. User action ŌåÆ Next.js Server Action or API Route
2. Server Action calls Service ŌåÆ Service calls Repository ŌåÆ Repository calls Supabase
3. Service emits `system_event` record to Postgres
4. Inngest function triggers on event (via `inngest.send()`) or polls
5. Inngest job creates `workflow_run` + `job_execution` records
6. Job may create `approval_request` ŌåÆ pauses until resolved
7. On completion, job updates `workflow_run.status`

### 3. MULTI-TENANCY MODEL

#### Isolation Strategy

Every tenant-owned table includes `tenant_id`. RLS policies ensure a user can only access rows belonging to their current tenant. No row from one tenant is ever visible to another, even through joins.

**JWT Custom Claims approach:**
When a user signs in, a Supabase Auth hook (Edge Function `on_auth_token_created`) populates JWT claims:
```json
{
  "sub": "<user-uuid>",
  "app_metadata": {
    "tenant_id": "<tenant-uuid>",
    "workspace_id": "<workspace-uuid>",
    "role_slug": "member"
  }
}
```

The active workspace is set at login time (defaulting to the user's primary workspace) and can be switched mid-session by calling a server action that re-issues the session with updated `app_metadata`.

**RLS Helper Functions:**
```sql
-- Extract tenant from JWT
CREATE OR REPLACE FUNCTION auth.tenant_id() RETURNS uuid AS $$
  SELECT (auth.jwt() -> 'app_metadata' ->> 'tenant_id')::uuid
$$ LANGUAGE sql STABLE;

-- Extract workspace from JWT
CREATE OR REPLACE FUNCTION auth.workspace_id() RETURNS uuid AS $$
  SELECT (auth.jwt() -> 'app_metadata' ->> 'workspace_id')::uuid
$$ LANGUAGE sql STABLE;

-- Check if user is tenant admin
CREATE OR REPLACE FUNCTION auth.is_tenant_admin() RETURNS boolean AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships m
    JOIN roles r ON r.id = m.role_id
    WHERE m.user_id = auth.uid()
      AND m.tenant_id = auth.tenant_id()
      AND r.slug IN ('tenant_admin', 'platform_admin')
      AND m.status = 'active'
  )
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

**Standard RLS policy pattern (applied to every tenant-owned table):**
```sql
-- tenant isolation
CREATE POLICY "tenant_isolation" ON leads
  FOR ALL USING (tenant_id = auth.tenant_id());

-- workspace membership (for tables with workspace_id)
CREATE POLICY "workspace_membership" ON leads
  FOR ALL USING (
    workspace_id IN (
      SELECT workspace_id FROM memberships
      WHERE user_id = auth.uid() AND status = 'active'
    )
  );
```

### 4. SECURITY / AUTHORIZATION MODEL

#### Auth Flow
1. User registers/signs in via Supabase Auth (email/password or magic link)
2. `on_auth_token_created` Edge Function runs ŌåÆ looks up membership ŌåÆ adds `tenant_id`, `workspace_id`, `role_slug` to JWT `app_metadata`
3. Next.js middleware validates session on every protected route using `@supabase/ssr`
4. Server components/actions use `createServerClient` (cookie-based) ŌĆö never the anon client for protected data
5. The service role client (`createServiceClient`) is used ONLY in Inngest handlers and webhook receivers ŌĆö never exposed to the browser

#### Role Model (System Roles)
| Role Slug | Scope | Capabilities |
|-----------|-------|--------------|
| `platform_admin` | platform | Full access across all tenants |
| `tenant_admin` | tenant | Full access within their tenant |
| `workspace_admin` | workspace | Full access within their workspace |
| `member` | workspace | Standard CRUD on assigned modules |
| `viewer` | workspace | Read-only access |

Custom roles can be created per tenant and assigned any combination of permissions.


#### Lifecycle Models

**workflow_run lifecycle:**
```
pending ŌåÆ running ŌåÆ [paused (approval)] ŌåÆ completed
                 Ōåś                      Ōåś failed
                  cancelled
```

**job_execution lifecycle:**
```
pending ŌåÆ running ŌåÆ completed
                 Ōåś failed ŌåÆ (retry) ŌåÆ running...
                 Ōåś skipped
```

**approval_request lifecycle:**
```
pending ŌåÆ approved ŌåÆ (workflow resumes)
       Ōåś rejected ŌåÆ (workflow cancelled or alternative path)
       Ōåś expired  ŌåÆ (policy-defined behavior)
```

#### Idempotency
- `system_events` has a `UNIQUE` constraint on `idempotency_key`
- Inngest uses `event.id` as its own idempotency key
- All Inngest functions must be written as idempotent operations
- Use `ON CONFLICT DO NOTHING` when recording events from webhooks

#### Retry Strategy
- Inngest handles retries natively (exponential backoff)
- `job_executions.attempt` tracks retry count
- `automation_failures` is created after max retries exceeded

### 6. ARTIFACT / KNOWLEDGE MODEL

**Storage layout in Supabase Storage:**
```
bucket: artifacts
  /{tenant_id}/{subject_type}/{subject_id}/{artifact_id}/{version_number}/{filename}
```

**Upload flow:**
1. Client requests signed upload URL via server action
2. Server action validates permission, creates `artifact` record (status: `processing`)
3. Client uploads directly to Supabase Storage
4. Client calls completion server action ŌåÆ updates artifact status to `active`
5. Server emits `artifact.uploaded` system event
6. Inngest function picks up event ŌåÆ creates `document_extraction` record (Phase 3 fills this)

**Retrieval flow:**
1. Server generates signed download URL (short TTL, 60 min)
2. Never expose raw Storage URLs to client

#### Prompt Template Variable System
Prompts use `{{variable_name}}` syntax. The `variables` field on `prompt_versions` declares what context variables the prompt expects. The caller is responsible for providing a complete context object.

### 8. PRODUCTIZATION / GTM READINESS MODEL

#### Tenant Onboarding Model
1. Platform admin creates tenant record + default workspace
2. First user is assigned `tenant_admin` role
3. System applies `industry_profile` defaults to scaffold their configs
4. Tenant admin configures sender identity + domain
5. Feature entitlements gate advanced capabilities

#### Industry Pack Model
Each `industry_profiles` record defines defaults for an entire vertical:
- Pipeline stage configs
- Default scoring dimensions
- Default workflow triggers
- Default prompt templates
- Feature flag defaults

New vertical = new `industry_profiles` seed record + associated default configs.

#### White-Label Path
`branding_profiles` + `domain_settings` support custom domains per tenant. Future: deploy tenant on custom subdomain (`acme.verian.ai`) via Next.js `host` header routing and middleware workspace resolution.

#### Evolution Path
```
Phase 1-2: Internal BIOS (single tenant, no billing)
Phase 3-4: Multi-tenant managed platform (invite-only)
Phase 5+:  Self-serve SaaS with billing (Stripe entitlements)
```

## PHASE 1 ŌĆö KEY DESIGN DECISIONS & RATIONALE

| Decision | Rationale |
|----------|-----------|
| Modular monolith over microservices | Faster to build, easier to debug, can extract services later |
| Separate score tables per score type | Clean indexes, separate scoring configs, avoid type-juggling |
| `system_events` as immutable log | Inngest triggers from events; audit log is a side effect |
| JWT custom claims for tenant_id | Avoids per-request DB lookup in every RLS policy |
| `idempotency_key` on system_events | Prevents duplicate workflow triggers from retried requests |
| Server Actions over API routes for mutations | Type-safe, no serialization layer, simpler auth |
| Service role only in Inngest/webhooks | Never expose service role to browser; strict boundary |
| `industry_profiles` as default seeds | Enables multi-vertical without code changes |
| Prompt configs in DB, not code files | Tenant-overridable, versioned, auditable |

## RISK AREAS

1. **JWT claim staleness** ŌĆö if tenant/workspace changes, JWT must be refreshed. Middleware must handle token refresh carefully.
2. **RLS performance** ŌĆö complex RLS policies with JOINs can slow queries. Profile early; consider denormalized `tenant_id` on every table (already planned).
3. **Inngest cold start** ŌĆö on Vercel, Inngest needs to be able to reach `/api/inngest`. Ensure this route isn't behind auth middleware.
4. **Supabase Storage signed URLs** ŌĆö keep TTLs short (60 min max). Don't cache signed URLs across sessions.
5. **Webhook signature validation** ŌĆö Resend webhooks MUST validate signatures before processing to prevent spoofed email events.
